# secret_com 授权系统——安全性深度解析

> 版本 1.0 · 2026 年 3 月  
> 适用读者：公司内部技术评审、安全审计、决策层

---

## 一、整体安全威胁模型

系统针对以下 **6 类攻击面** 设计防御，各层相互独立，构成完整纵深防御体系：

```
┌──────────────────────────────────────────────────────────────────────┐
│                         攻击面全景                                    │
│                                                                      │
│  攻击者身份         攻击手段                      防御层              │
│  ──────────         ────────                      ────              │
│  串口监听者    →  被动窃听串口报文           →  AES-256-GCM 全程加密  │
│  中间人        →  篡改/注入消息             →  GCM 认证标签 + ECDSA  │
│  Token 重放者  →  复用历史合法报文           →  临时密钥 + nonce + 时间戳 │
│  Token 伪造者  →  自制假 Token             →  ECDSA-P256 签名验证   │
│  固件逆向者    →  从电调 dump 固件提取"密钥"→  电调仅持有公钥（非机密）│
│  RK3506 克隆者 →  物理拆解提取私钥          →  TPM/OTP/OP-TEE 安全存储│
└──────────────────────────────────────────────────────────────────────┘
```

## 二、四层纵深防御结构

### 第 0 层：非对称密钥分布——信任根

这是整个安全体系最根本的基础。私钥绝不进入电调，从架构上消除了"一台被攻破 → 全部失效"的风险。

```
┌──────────────────────────────────────────────────────────────────┐
│                        密钥分布设计                               │
│                                                                  │
│   公司后台              RK3506（授权主控）         电调（ESC）    │
│   ──────               ──────────────────         ──────────    │
│   生成密钥对             持有：签名私钥（32 B）      持有：验证公钥 │
│   管理密钥副本 ─────────► 存储：TPM / OTP / OP-TEE  （65 B）     │
│                                                   ← 非机密信息  │
│                         ↓ ECDSA-Sign              ↓ ECDSA-Verify│
│                         颁发 Token                接受 / 拒绝    │
└──────────────────────────────────────────────────────────────────┘

核心推论：
  即使攻击者通过 JTAG/UART 完整读取电调固件，
  仅能获得公钥 —— 无法用公钥伪造 Token
  ∵  ECDSA 非对称性：签名需私钥，验证只需公钥
  ∴  所有电调固件泄露 ≠ 授权体系失效
```

**与对称密钥方案的本质区别：**
传统方案将 AES 密钥烧入设备 → 一台被攻破则同型号全部失效。
本方案电调仅有公钥 → 任意数量电调被攻破 → 攻击者零收益。

---

### 第 1 层：ECDH 临时密钥协商——前向保密

每次上电都生成全新的临时密钥对，历史会话密文永远无法被解密。

```
每次上电握手流程（前向保密原理）：

RK3506                                              电调（ESC）
  │                                                     │
  │  ① 生成一次性临时密钥对                              │
  │     (rk_eph_priv, rk_eph_pub) ← CSPRNG              │
  │     生成 rk_random (16 B)                            │
  │                                                     │
  │──── HELLO_REQ [0x01] ──────────────────────────────►│
  │     rk_eph_pub (65 B) + rk_timestamp + rk_random    │
  │                                                     │  ② 生成一次性临时密钥对
  │                                                     │     (esc_eph_priv, esc_eph_pub)
  │                                                     │     生成 esc_random (16 B)
  │◄─── HELLO_RSP [0x02] ──────────────────────────────│
  │     esc_eph_pub (65 B) + esc_random                 │
  │                                                     │
  │  ③ 双方独立计算（数学保证结果相同）                    │
  │                                                     │
  │  shared = ECDH(rk_eph_priv, esc_eph_pub)            │  shared = ECDH(esc_eph_priv, rk_eph_pub)
  │                                                     │
  │  session_key = HKDF-SHA256(                         │  session_key = HKDF-SHA256(
  │      IKM  = shared,                                 │      IKM  = shared,
  │      salt = rk_random || esc_random,                │      salt = rk_random || esc_random,
  │      info = "secret_com_session_key_v1"             │      info = "secret_com_session_key_v1"
  │  )                                                  │  )
  │                                                     │
  │  ④ 临时私钥立即销毁                                  │  临时私钥立即销毁
  │     ↓                                               │
  │     历史报文永远无法被解密（前向保密 PFS）              │
```

**前向保密的实际意义：**

```
攻击场景对比：

  RSA 静态密钥方案（无 PFS）：
    攻击者录制全部历史通信 ──► 某天私钥泄露 ──► 解密所有历史报文 ✗

  ECDH 临时密钥（有 PFS）：
    攻击者录制全部历史通信 ──► 某天私钥泄露 ──► 仅影响未来会话
                                                 历史临时私钥已销毁
                                                 历史报文永远无法解密 ✓
```

### 第 2 层：AES-256-GCM 加密传输——防窃听 & 防篡改

Token 在串口上全程以密文传输，且携带认证标签防止任何篡改。

```
TOKEN_PUSH [0x03] 帧结构：

  ┌─────────────────────────────────────────────────────────────────┐
  │  帧头：[0xAA][0x55][0x03][SeqNum:2 LE][PayLen:2 LE]            │
  ├─────────────────────────────────────────────────────────────────┤
  │  Payload:                                                       │
  │  ┌───────────────────────────────────────────────────────────┐  │
  │  │  nonce     (12 B, 随机生成，每次不同)                      │  │
  │  ├───────────────────────────────────────────────────────────┤  │
  │  │  ciphertext = AES-256-GCM(                                │  │
  │  │      key   = session_key,       ← 本次握手派生的临时密钥   │  │
  │  │      nonce = 上方 12 B nonce,   ← 每次随机，防重放         │  │
  │  │      plain = Token 明文          ← 60 B 固定 + ≤72 B 签名  │  │
  │  │  )                                                        │  │
  │  ├───────────────────────────────────────────────────────────┤  │
  │  │  GCM-tag   (16 B, 认证标签)                               │  │
  │  │  ← 密文任意 1 bit 被修改 → tag 验证失败 → 消息被丢弃       │  │
  │  └───────────────────────────────────────────────────────────┘  │
  ├─────────────────────────────────────────────────────────────────┤
  │  CRC16-CCITT (2 B, 覆盖整帧)                                    │
  └─────────────────────────────────────────────────────────────────┘
```

**AES-GCM vs AES-CBC 的关键差异：**

| 特性 | AES-256-CBC | AES-256-GCM |
|------|:-----------:|:-----------:|
| 加密（机密性） | ✅ | ✅ |
| 认证（完整性） | ❌ | ✅ |
| 防 bit-flipping 篡改 | ❌ | ✅ |
| 防 Padding Oracle 攻击 | ❌（需额外处理）| ✅（无 Padding）|
| 并行化计算 | ❌ | ✅ |
| 现代密码学标准推荐 | 逐渐淘汰 | ✅ NIST 推荐 |

### 第 3 层：Token 内置 ECDSA 签名——最终防伪造防线

即使前三层全部被绕过，Token 本身还有独立的密码学签名保护，且签名覆盖所有字段。

```
Token 签名计算（RK3506 侧）：

  signed_data = SHA-256(
      "secret_com_token_v1\0"   ← 域分隔符（20 B，防跨协议签名复用）
      || esc_device_id           ← 绑定目标设备（4 B）
      || feature_flags           ← 绑定功能权限（1 B）
      || license_type            ← 绑定授权类型（1 B）
      || padding                 ← 对齐（2 B）
      || issue_timestamp         ← 时间绑定（8 B）
      || expiry_timestamp        ← 有效期绑定（8 B）
      || remaining_uses          ← 次数绑定（4 B）
      || product_name            ← 产品绑定（32 B）
  )   ——→ 32 B 哈希值（共 80 B 输入）

  signature = ECDSA-P256-Sign(rk3506_private_key, signed_data)
              ↑
              唯一持有者：RK3506 安全存储区

  ──────────────────────────────────────────────────────
  电调验证流程（无私钥参与）：

  ① AES-GCM 解密 TOKEN_PUSH 密文
  ② ECDSA-P256-Verify(esc_builtin_public_key, signed_data, signature)
  ③ 校验 token.esc_device_id == my_device_id   ← 防设备间复用
  ④ 校验 issue_timestamp / expiry_timestamp     ← 防时间攻击
  ⑤ 全部通过 → g_authorized = 1，允许电机工作
     任意失败 → 拒绝启动
```

**域分隔符（"secret_com_token_v1\0"）的作用：**
防止攻击者将其他协议产生的合法 ECDSA 签名（如 TLS 握手、JWT 等）拼接至 Token 中伪装成合法签名，这是密码学实现中常被忽视的关键细节。

## 三、算法选型详细论证

### 3.1 ECDSA P-256 vs RSA

| 维度 | ECDSA P-256 | RSA-2048 | RSA-3072 |
|------|:-----------:|:--------:|:--------:|
| 等效安全强度 | **~128 bit** | ~112 bit | ~128 bit |
| 私钥大小 | **32 B** | 1232 B | 1847 B |
| 公钥大小（固化进固件） | **65 B** | 294 B | 422 B |
| 签名大小 | **≤72 B** | 256 B | 384 B |
| 验签速度（Cortex-M4 @168MHz）| **~200 ms** | >2000 ms | >5000 ms |
| Flash 占用（mbedTLS 子集）| **~20 KB** | ~40 KB | ~40 KB |
| STM32 PKA 硬件加速支持 | **✅ 原生** | ❌ | ❌ |
| FIPS 140-2/3 合规 | **✅** | ✅ | ✅ |

**结论：** 对于 Cortex-M4/M7 嵌入式 MCU，P-256 在 128-bit 安全强度下提供最紧凑的密钥尺寸和最快的验签速度，RSA 在嵌入式场景代价过高。若使用 STM32H7 的 PKA 硬件加速，P-256 验签可进一步压缩至 ~10ms。

---

### 3.2 ECDH 临时密钥 vs 静态 RSA 加密

```
静态 RSA 加密会话密钥（无前向保密）：

  攻击者持续录制所有串口通信 ─────────────────────────────────────┐
                                                                   │
  某天 RK3506 被物理攻破，私钥泄露 ─────────────────────────────┐  │
                                                                │  │
                     攻击者用私钥解密所有历史会话密钥 ←──────────┘  │
                     再解密所有历史 Token 报文 ←───────────────────┘
                     → 历史所有授权记录全部暴露 ✗

ECDH 临时密钥（有前向保密 PFS）：

  攻击者持续录制所有串口通信 ─────────────────────────────────────┐
                                                                   │
  某天 RK3506 被物理攻破，私钥泄露 ─────────                       │
                                        │                         │
                      仅影响当前及未来会话 │                        │
                      历史临时私钥已销毁  │                        │
                      历史报文永远无法解密 ←────────────────────────┘
                                                    ✓ PFS 保护
```

ECDH 的数学基础：椭圆曲线 Diffie-Hellman 允许双方在不传输任何私钥材料的情况下协商出相同密钥。中间人只能观察到两个公钥，无法推导 shared secret（基于椭圆曲线离散对数困难问题 ECDLP）。

---

### 3.3 HKDF-SHA256 密钥派生

ECDH 原始输出不能直接用于 AES：

```
ECDH 原始输出问题：
  shared_secret = P_x   （P-256 曲线上某点的 X 坐标）
                  ↓
                  非均匀分布，不是密码学意义上的随机密钥
                  直接用于 AES → 存在统计偏差安全隐患

HKDF-SHA256 处理（RFC 5869）：

  Step 1 (Extract):
    PRK = HMAC-SHA256(
        key  = rk_random || esc_random,   ← 双方随机数作盐，消除 ECDH 偏差
        data = shared_secret
    )   → 均匀随机的伪随机密钥材料

  Step 2 (Expand):
    session_key = HMAC-SHA256(
        key  = PRK,
        data = "secret_com_session_key_v1" || 0x01
    )   → 32 B，密码学意义上的均匀随机密钥

  额外安全性：
    info = "secret_com_session_key_v1" 绑定使用场景
    防止同一 shared_secret 在不同上下文派生出相同密钥
```

---

### 3.4 AES-256-GCM vs 其他加密模式

| 模式 | 加密 | 认证 | 抗篡改 | Padding | 嵌入式性能 | 推荐度 |
|------|:----:|:----:|:------:|:-------:|:----------:|:------:|
| AES-256-ECB | ✅ | ❌ | ❌ | 需要 | 高 | ❌ 不安全 |
| AES-256-CBC | ✅ | ❌ | ❌ | 需要 | 高 | ⚠️ 淘汰中 |
| AES-256-CTR | ✅ | ❌ | ❌ | 不需要 | 高 | ⚠️ 需配合 MAC |
| AES-256-GCM | ✅ | ✅ | ✅ | 不需要 | 高（可并行）| ✅ **推荐** |
| ChaCha20-Poly1305 | ✅ | ✅ | ✅ | 不需要 | 高（无 AES 加速时更快）| ✅ 备选 |

**选 AES-256-GCM 的理由：**
1. NIST SP 800-38D 标准，FIPS 140-2/3 认可
2. STM32 等 MCU 内置 AES 硬件加速，GCM 可利用硬件
3. mbedTLS 内置完整支持，无需额外依赖
4. 单次 API 调用同时完成加密和认证，降低实现出错概率

---

### 3.5 P-256 vs Curve25519

| 维度 | P-256 (secp256r1) | Curve25519 |
|------|:-----------------:|:----------:|
| 安全强度 | ~128 bit | ~128 bit |
| mbedTLS 完整内置 | ✅ | 需额外配置 |
| STM32 PKA 硬件加速 | ✅ 原生支持 | ❌ 多数型号不支持 |
| FIPS 140-2/3 认证 | ✅ 合规 | ❌ 不在 FIPS 列表 |
| 工业/商业合规场景 | ✅ | ❌（部分受限）|
| 跨平台库支持广度 | ✅ 极广 | 良好但不及 P-256 |

对于工业电调产品，P-256 是最广泛支持且合规的选择，在商业合规要求较高的客户场景下更具优势。

## 四、安全属性量化评估

| 安全属性 | 评级 | 技术依据 |
|---------|:----:|---------|
| **Token 不可伪造性** | 🔒🔒🔒 极强 | 破解需解决 P-256 ECDLP，当前计算不可行（128-bit 安全）|
| **传输机密性** | 🔒🔒🔒 极强 | AES-256-GCM，密钥空间 $2^{256}$ |
| **前向保密性（PFS）** | 🔒🔒🔒 极强 | 每次握手独立临时密钥，历史报文永远无法解密 |
| **重放攻击防御** | 🔒🔒🔒 极强 | 临时密钥对 + 随机 nonce + 时间戳三重绑定 |
| **传输完整性** | 🔒🔒🔒 极强 | AES-GCM tag（128-bit），任意修改均被检测 |
| **设备间 Token 复用防御** | 🔒🔒🔒 极强 | Token 内嵌 `esc_device_id`，电调强制校验一致性 |
| **固件逆向防御** | 🔒🔒🔒 极强 | 电调仅持有公钥，逆向固件对攻击者零增益 |
| **中间人（MITM）防御** | 🔒🔒 强 | GCM tag + ECDSA 双重认证；HELLO 阶段无签名（见风险说明）|
| **RK3506 物理攻击防御** | 🔒🔒 强 | 依赖安全存储介质（TPM/OTP/OP-TEE）的实现质量 |
| **时钟伪造攻击防御** | 🔒 中 | 依赖电调 RTC 精度；可通过令 Token 有效期短且频繁续签缓解 |

## 五、残余风险与缓解措施

### 5.1 RK3506 物理安全（最高优先级）

RK3506 持有的签名私钥是整个体系的唯一信任根，物理安全是最终防线。

**私钥保护层次（由强到弱）：**

```
级别 1（推荐）：OP-TEE 安全世界 + 硬件密钥包裹
  ─ 私钥从不以明文形式出现在普通内存
  ─ 只有 TEE 可信执行环境可调用签名操作
  ─ 即使 Linux 内核被攻破，私钥仍不可提取

级别 2：Linux TPM 2.0 密封存储
  ─ 私钥由 TPM 硬件保护
  ─ 需 PCR 度量值（系统完整性）授权方可使用
  ─ 设备被物理拆解后，PCR 值变化，密钥无法解封

级别 3：加密 Flash 分区（LUKS + TPM 密封解密密钥）
  ─ 私钥存于 LUKS 加密分区
  ─ 解密密钥由 TPM 密封，防离线暴力破解

级别 4（⚠️ 不推荐）：普通文件系统明文存储
  ─ 仅适用于实验室原型阶段
  ─ 不得用于出厂量产设备
```

### 5.2 HELLO 握手阶段无认证问题

当前设计中，HELLO_REQ 消息（RK3506 的临时公钥）**未携带签名**，理论上存在主动 MITM 插入假公钥的可能。

**实际风险评估：**
- 攻击者需物理接入串口并实时中继，门槛极高
- 即使 MITM 成功建立加密信道，也无法伪造最终 Token 签名
- Token 验签失败会立即暴露中间人存在

**可选加强措施：**
在 HELLO_RSP 中增加电调对 RK3506 临时公钥的签名验证。但这要求电调持有一个身份静态密钥对，增加了电调端的密钥管理复杂度，与"电调少做事"原则有所冲突，需权衡后决定。

### 5.3 时间源可靠性

订阅授权模式（kTimeLimited）依赖电调 RTC 的时间准确性。攻击者若能修改 RTC（如通过物理拆解操作），可延长 Token 有效期。

**缓解措施：**
- 使用 RTC 防篡改功能（STM32 RTC Tamper 引脚）
- 配置 RTC 电池备份，防断电归零
- Token 内的 `issue_timestamp` 可作为辅助基准（当前时间不应早于颁发时间）

## 七、总结

本系统在密码学选型上均采用工业界最佳实践，安全强度达到 **128-bit 对称密钥等效安全级别**，构建了 4 层独立纵深防御。

**核心安全保证链：**

```
私钥永不离开 RK3506 安全存储
    ↓
```
