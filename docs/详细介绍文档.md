# secret_com 密码学安全架构与算法详解

在我们的授权模型中，**RK3506 作为授权服务端（AuthServer/签发端），ESC（电调）作为验证客户端（AuthClient/验证端）**。为了确保这一架构在资源受限且暴露的嵌入式环境中做到绝对安全，本系统采用了经过业界广泛验证的“ECDH + AES-GCM + ECDSA”组合体系。

以下将详细解答 *为什么要使用这三种算法*，以及它们如何在嵌入式环境中对抗常见的安全威胁。

---

## 1. ECDH (P-256) —— 临时密钥安全交换

**算法作用：** 在 RK3506 和 ESC 之间动态协商出一个共享会话密钥（Session Key）。
**为何在嵌入式环境使用：** 
- **前向安全性（Forward Secrecy）**：每次设备重启或重新连接时，双方都会生成全新的临时椭圆曲线密钥对。通过交换临时公钥，双方在本地独立计算出同一个共享密钥。
- **防止提取密钥**：由于共享密钥**从未在串口或线缆中明文传输**，即使攻击者截获了所有的串口通讯包（如使用逻辑分析仪），或者反编译固件提取到了静态代码，也无法通过截获的报文推导出当前的会话密钥，保证了即使在暴露的物理总线上也能安全建立连接。

## 2. AES-256-GCM —— 认证加密隧道防御

**算法作用：** 对实际的授权请求包与响应包（License）进行数据加密及完整性保护。
**为何在嵌入式环境使用：**
- **高吞吐性能与并行度**：AES-GCM 是基于计数器模式（CTR）的加密算法，附带伽罗瓦消息认证码（GMAC）。它在许多包含硬件级 AES 指令或加密协处理器的 MCU/SoC（如 RK3506 和高级 ESC 芯片）上运行极快，性能远超于持续进行非对称加解密。
- **防篡改与防中间人控制（MITM）**：AES-GCM 在加密数据的同时会生成消息认证码（MAC）。如果攻击者在传输路径上篡改哪怕一个字节的密文，GCM 的认证标签校验就会失败，硬件设备会立刻切断连接并报警。
- **防御重播攻击（Replay Attack）**：结合每次握手过程中生成的随机数（Nonce）与递增序列号，任何被录屏、被抓取的“已授权”数据包在下一次会话甚至同一会话的下一次报文中发送都会失效，保证黑客无法通过“录制以往合法启动发出的数据”来欺骗电调执行解锁。

## 3. ECDSA (P-256) —— 基于数字签名的不可伪造授权

**算法作用：** 保证授权的根信任度。RK3506 使用其保护在硬件里（如 OTP / Secure Flash）的私钥对授权凭证进行签名；ESC 使用固化在闪存中的公钥来验证这个签名。
**为何在嵌入式环境使用：**
- **防克隆与防伪造（Anti-Cloning）**：仅仅有加密隧道是不够的。任何可以拆解设备的人都可以获取 ESC 里的对称密钥进行逆向。但利用 ECDSA 的非对称特性，ESC 内部被提取出来的仅仅是**公钥**，公钥仅能用来“验证”而无法用来“签发”。如果黑客想要制作一个仿冒的 RK3506（例如用另一块开发板发送虚假授权信号），由于他没有官方的私钥文件，根本无法计算出合法的 ECDSA 签名，ESC 会立即识破并拒绝解锁。
- **离线与去中心化验证**：在很多无人机、机器人等野外作业场景中没有外网，系统必须实现纯本地验证。ECDSA 兼具体积小、签名短的特点（P-256 签名仅占 64 个字节左右），能在不对嵌入式设备造成明显网络延迟和存储压力的前提下，达到银行级的抗篡改安全标准。

## 总结：联防联动的授权闭环

| 潜在攻击流 | 系统防御策略 | 拦截依靠 |
| :--- | :--- | :--- |
| **设备伪造/山寨克隆** | 伪造的主板无法提供合法私钥生成的 ECDSA 签名 | **ECDSA** |
| **串口抓包重放旧授权**| 每轮通信 Nonce 均不同，录制包无法通过 MAC 校验 | **AES-GCM** |
| **暴力破解总线上的密钥**| 密钥通过 ECDH 协商且不在链路上明码传播 | **ECDH** |
| **恶意的中间人报文篡改**| GCM 认证标签（Tag）强校验，一字节不同即被丢弃 | **AES-GCM** |

通过“**RK3506 负责签发授权（签署 ECDSA），ESC 负责验证并执行授权**”的设计架构，并配合上述三大密码学基石，本方案能够从物理层到逻辑层全面抵御嵌入式环境下极具危险性的多种硬件级攻击。
