> For ESC device developers and client-application developers.  
> Version 1.0 · March 2026

## Part A — ESC Device Developer (Server Side)

### 1. Key Pair Generation

Generate a P-256 key pair **once per device model** (or per device for stronger isolation):

```bash
# Requires OpenSSL 1.1+
openssl ecparam -name prime256v1 -genkey -noout -out device.pem

# Export private key as raw DER, then extract 32-byte scalar
openssl ec -in device.pem -outform DER -out device_priv.der
# Bytes [7..38] of device_priv.der are the 32-byte private scalar.

# Export public key in uncompressed point form (65 bytes: 0x04 || X || Y)
openssl ec -in device.pem -pubout -outform DER -out device_pub.der
# Bytes [27..91] of device_pub.der are the 65-byte uncompressed public key.
```

Alternatively, use the provided `keygen` Python script:

```bash
python3 tools/keygen.py --out device_keys.h
```

This generates a ready-to-include C header with:
```c
static const uint8_t kDevicePublicKey[65]  = { ... };
static const uint8_t kDevicePrivateKey[32] = { ... };  // KEEP SECRET
```

### 2. Burning the Private Key

- **Bare-metal STM32 / NXP**: Store in OTP flash or a TZEN/TrustZone-protected region.  
- **Linux-based ESC**: Store in the Linux TPM or an encrypted keyring; never in plain text.  
- The public key is **not secret** — distribute it freely to client builds.

### 3. Integrating the Server into Firmware

```cpp
#include "secret_com/auth_server.h"
#include "src/transport/serial_transport.h"
#include "device_keys.h"  // generated by keygen.py

void AuthTask(void* /* pvParameters */) {
  auto transport = std::make_unique<secret_com::SerialTransport>(
      "/dev/ttyS1", 115200);

  secret_com::CryptoConfig cfg;
  cfg.is_server = true;
  memcpy(cfg.device_public_key,  kDevicePublicKey,  secret_com::kEcPublicKeySize);
  memcpy(cfg.device_private_key, kDevicePrivateKey, secret_com::kEcPrivateKeySize);

  secret_com::AuthServer server;
  server.Initialize(std::move(transport), cfg);

  server.SetAuthVerifyCallback(
      [](const secret_com::AuthRequest& req,
         secret_com::LicenseInfo*       lic) -> bool {
        // Query your CRM / allow-list database here.
        if (!db_is_licensed(req.client_device_id, req.product_name))
          return false;

        lic->type          = secret_com::LicenseType::kPermanent;
        lic->device_id     = req.client_device_id;
        lic->feature_flags = secret_com::kFeatureAll;
        strncpy(lic->product_name, req.product_name, 32);
        return true;
      });

  // Blocking loop (or use StartListening() for async).
  for (;;) {
    server.HandleOneRequest(30000 /* ms */);
  }
}
```

### 4. Bare-metal / FreeRTOS (no POSIX threads)

Use `HandleOneRequest()` from within your RTOS task.  
For I/O, use `CallbackTransport` instead of `SerialTransport`:

```cpp
#include "secret_com/transport.h"

secret_com::CallbackTransport::Callbacks cbs;
cbs.connect      = []{ return uart_open(UART2) == HAL_OK; };
cbs.disconnect   = []{ uart_close(UART2); };
cbs.is_connected = []{ return uart_is_ready(UART2); };
cbs.send = [](const uint8_t* d, size_t n) {
    return (int)uart_transmit_blocking(UART2, d, n, 1000);
};
cbs.receive = [](uint8_t* b, size_t n, uint32_t t) {
    return (int)uart_receive_timeout(UART2, b, n, t);
};
auto transport = std::make_unique<secret_com::CallbackTransport>(cbs);
```

### 5. Disable Unused mbedTLS Features (embedded)

Add to your `mbedtls_config.h` to minimize flash usage:

```c
#define MBEDTLS_ECP_C
#define MBEDTLS_ECDH_C
#define MBEDTLS_ECDSA_C
#define MBEDTLS_AES_C
#define MBEDTLS_GCM_C
#define MBEDTLS_SHA256_C
#define MBEDTLS_HKDF_C
#define MBEDTLS_CTR_DRBG_C
#define MBEDTLS_ENTROPY_C
#define MBEDTLS_ECP_DP_SECP256R1_ENABLED
/* Disable everything else */
```

## Part B — Client Application Developer

### 1. Embedding the Device Public Key

Obtain `kDevicePublicKey` from the ESC manufacturer.  Embed it at build time:

```cpp
// In a header signed with your code-signing key.
static const uint8_t kEscPublicKey[secret_com::kEcPublicKeySize] = {
  0x04,
  // 32 bytes X ...
  // 32 bytes Y ...
};
```

Never ship the private key in client code.

### 2. C++ Client Initialization

```cpp
#include "secret_com/auth_client.h"
#include "src/transport/serial_transport.h"

secret_com::CryptoConfig cfg;
memcpy(cfg.device_public_key, kEscPublicKey, secret_com::kEcPublicKeySize);
cfg.is_server = false;

auto transport = std::make_unique<secret_com::SerialTransport>(
    GetSerialPort(), 115200);

secret_com::AuthClient client;
auto s = client.Initialize(std::move(transport), cfg);
// s == Status::kOk on success.
```

### 3. Requesting Authorization

```cpp
secret_com::AuthRequest req;
req.client_device_id = ReadDeviceUniqueId();  // e.g. STM32 UID
strncpy(req.product_name, "ESC-Pro-X1", 32);
req.feature_flags = secret_com::kFeatureAll;

client.RequestAuthorization(req, [](secret_com::Status s,
                                     const secret_com::LicenseInfo& lic) {
    if (s == secret_com::Status::kOk) {
        // Unlock ESC functionality.
        esc_enable_full_mode();
    } else {
        // Show error / run in demo mode.
        show_error(StatusToString(s));
    }
}, 10000 /* 10 s timeout */);
```

### 4. License Persistence

After a successful authorization, serialize the license for caching:

```cpp
// Persist (use your platform's non-volatile storage).
auto lic = client.GetLicenseInfo();
nvs_write("license", &lic, sizeof(lic));

// On next boot, verify the cached license before requesting network auth.
LicenseInfo cached;
if (nvs_read("license", &cached, sizeof(cached)) == 0) {
    if (cached.IsValid(get_unix_time())) {
        // Continue without re-authorization.
    }
}
```

**Important**: A cached license token does not replace the online authorization for the first use — the signature is verified against the device public key on every call to `ReceiveAuthResponse()`.

### 5. C API (bare-metal / C projects)

```c
#include "secret_com/secret_com.h"

SecretComIoCbs io = {
  .user_ctx    = NULL,
  .connect     = my_uart_connect,
  .disconnect  = my_uart_disconnect,
  .is_connected= my_uart_is_connected,
  .send        = my_uart_send,
  .receive     = my_uart_receive,
};

SecretComClient* client = SecretComClientCreate();
SecretComClientInit(client, &io, kEscPublicKey);

SecretComAuthRequest req = {
  .client_device_id = 0x12345678,
  .product_name     = "ESC-Pro-X1",
  .feature_flags    = 0xFF,
};

SecretComClientRequestAuth(client, &req, on_auth_done, NULL, 10000);
// ... wait for callback via RTOS event flag or semaphore
```

## Part C — Security Checklist

- [ ] Private key stored in OTP / HSM / TrustZone — NOT in plain flash
- [ ] `device_private_key` zero-filled in RAM after use (done automatically by auth_server.cc)
- [ ] Session key zero-filled after each authorization (done automatically)
- [ ] Client embeds public key in read-only flash segment (`const` + linker section)
- [ ] Firmware update mechanism validates code signature before replacing firmware
- [ ] Serial port hardware disabled when not in use (GPIO control)
- [ ] Timestamps validated: reject requests > 60 s old
- [ ] Allow-list / CRM database maintained server-side; not stored on client

## Part D — FAQ

**Q: Can the authorization code be replayed?**  
A: No. Each session uses a fresh ephemeral ECDH key pair and random nonces.  Captured traffic from one session cannot authorize another.

**Q: What if the serial port is tapped with a logic analyzer?**  
A: All payload data after the key exchange is AES-256-GCM encrypted with a session key derived from an ephemeral ECDH exchange.  The session key is never transmitted; it is derived independently on both sides.

**Q: What if the client binary is reverse-engineered and the embedded public key is extracted?**  
A: The public key is intentionally not secret.  An attacker who has the public key still cannot sign license tokens without the corresponding private key, which resides only on the ESC device.

**Q: Can two devices share a key pair?**  
A: It is supported but reduces isolation.  For maximum security, generate a unique key pair per device and use per-device certificate chains.

**Q: How do I revoke a license?**  
A: Remove the device ID from the server's allow-list.  The next time the client connects for re-authorization it will be denied.  Cached licenses can be invalidated by issuing a firmware update that clears the cache.
