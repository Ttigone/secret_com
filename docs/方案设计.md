## 一、背景与动机

### 1.1 业务背景

公司主要产品为**电调设备（ESC）**。当前存在以下业务痛点：

- 电调固件容易被仿制厂商复制，无法区分正版/仿版
- 无法按功能分级销售（基础版 / 高级版 / 完整版）
- 无法支持订阅制商业模式（按月/年收费）
- 无法远程吊销已出售设备的授权（如客户违约、拖欠费用）

### 1.2 设计约束

| 约束 | 说明 |
|------|------|
| 电调已量产 | 固件改动必须极小，不能破坏现有功能 |
| 资源受限 | Cortex-M4/M7，Flash/RAM 有限 |
| 无网络连接 | 电调不联网，不能依赖在线验证 |
| 低成本 | 授权硬件成本需控制 |
| 可维护性 | 授权策略变更不应要求重刷电调固件 |

### 1.3 核心设计原则

> **电调少做事，RK3506 多做事。**

所有授权逻辑、密码学运算、白名单管理集中在 RK3506 上实现，电调仅做最轻量的验签操作。这样，授权策略的任何升级只需更新 RK3506，无需触碰已量产电调固件。

> **电调绝不持有签名私钥。**

传统双向互认证方案（类 mTLS / SSH 公私钥体系）要求被认证设备也持有私钥，但电调是最容易被物理攻击的终端——固件一旦被 JTAG/SWD dump，私钥即宣告泄露。本方案将唯一的签名私钥集中在更易物理保护的 RK3506 上，电调仅持有公钥（公钥本身非机密），从架构上斩断了"固件逆向 → 私钥泄露 → 无限克隆授权"这一最高危攻击路径。详细对比论证见 §7.4。

### 2.2 角色职责划分

| 角色 | 持有密钥 | 主要职责 | 实现复杂度 |
|------|---------|---------|-----------|
| **电调（ESC）** | 验证公钥（65 B，非机密） | 验签 Token、门控业务逻辑 | 极低（~200 行 C）|
| **RK3506** | 签名私钥（32 B，受保护）| 生成 Token、管理白名单、接收远程指令 | 完整实现 |
| **公司后台** | 管理员权限、指令签名密钥 | 下发白名单、续期、撤销、密钥轮换 | Web 服务 |

### 4.4 永久授权模式——上电离线工作流程

```
电调上电
   │
   ├──► 读取 Flash 中的 Token
   │         │
   │    Token 存在？
   │    ├── YES ──► ECDSA 本地验签（无需联网/连接 RK3506）
   │    │               │
   │    │          验签通过？
   │    │          ├── YES ──► g_authorized = 1 ──► 正常工作
   │    │          └── NO  ──► 清除 Flash Token ──► 等待 RK3506
   │    │
   │    └── NO  ──► 等待 RK3506 串口连接推送 Token
   │                    │
   │               收到 Token ──► 验签 ──► 通过 ──► 写入 Flash
   │                                              ──► 正常工作
   │
   └──（每次上电均优先走离线验签路径，RK3506 只在首次激活时在线）
```

## 五、两种授权模式

### 5.1 模式对比总览

| 对比维度 | 模式一：永久买断 | 模式二：订阅续费 |
|---------|:--------------:|:--------------:|
| Token 类型 | kPermanent（0x00） | kTimeLimited（0x01）|
| `expiry_timestamp` | 0（永不过期）| 具体 Unix 时间戳 |
| Token 存入电调 Flash | ✅ 长期有效 | ✅ 但有有效期 |
| 每次上电需 RK3506 在线 | ❌ 仅首次 | Token 过期后需要 |
| 可远程撤销 | ❌（买断制本身不需要）| ✅ |
| 可立即强制撤销 | ❌ | ✅（推送撤销 Token）|
| 适用场景 | 一次性买断客户 | 订阅制、试用、租赁 |
| 电调额外需求 | Flash 存储 Token | Flash + 可靠 RTC |

### 5.2 模式一：永久买断详细说明

**适用场景：** 客户一次性付款购买，不需要续费，不需要吊销。

**核心特点：** Token 验签通过后写入电调内部 Flash，此后每次上电直接本地验签，**完全不依赖 RK3506 在线**。

**推荐订阅参数配置：**
```
license_type      = 0x00  (kPermanent)
expiry_timestamp  = 0     (永不过期)
feature_flags     = 按购买套餐配置
```

### 5.3 模式二：订阅续费详细说明

**适用场景：** 按月/年订阅，或公司需要保留吊销权的场景。

**核心特点：** Token 携带过期时间，电调每次上电检查有效期，过期后必须连接 RK3506 续期。

**推荐有效期配置：**

| 订阅类型 | 建议有效期 | 说明 |
|---------|:---------:|------|
| 月度订阅 | 35 天 | 略长于 30 天，容忍续期延迟 |
| 年度订阅 | 370 天 | 略长于 365 天 |
| 试用授权 | 14～30 天 | 功能受限（kFeatureBasic）|
| 永久但可撤销 | 3～5 年 | 长有效期但有上限 |

## 六、可行性分析

### 6.1 技术可行性

| 维度 | 评估 | 说明 |
|------|:----:|------|
| 嵌入式资源需求 | ✅ 可行 | 电调端：mbedTLS 子集约 30 KB Flash，~2 KB Stack |
| 开发工作量 | ✅ 低 | 电调端约 150～200 行 C 代码，接入 2 个 UART 回调即可 |
| 密码学运算速度 | ✅ 可接受 | P-256 验签在 Cortex-M4@168MHz 约 200ms；H7 PKA 加速可到 10ms |
| 跨平台兼容性 | ✅ 好 | C++14 标准，传输层可替换，支持 FreeRTOS/裸机/Linux |
| 依赖库成熟度 | ✅ 成熟 | mbedTLS 为 ARM 官方维护，工业界广泛使用 |
| 现有产品改造成本 | ✅ 低 | 对电调固件改动极小，不影响现有功能 |

### 6.2 生产可行性

| 维度 | 评估 | 说明 |
|------|:----:|------|
| 密钥生成 | ✅ 标准化 | OpenSSL 一条命令生成，有工具脚本 |
| 出厂写入流程 | ✅ 可行 | 私钥通过安全内网通道写入 RK3506；公钥编译进电调固件 |
| 设备 ID 录入 | ✅ 可行 | 生产线扫码录入或读取 MCU 芯片 UID 自动生成 |
| 白名单初始化 | ✅ 可行 | 按订单自动生成，写入 RK3506 本地数据库 |
| 成本影响 | ✅ 可接受 | 仅增加 RK3506 模块硬件成本 |

### 6.3 运营可行性

| 维度 | 评估 | 说明 |
|------|:----:|------|
| 客户体验 | ✅ 透明 | 授权过程对最终用户无感，上电自动完成 |
| 售后支持 | ✅ 简单 | 授权问题定位：查询 RK3506 日志 + TOKEN_ACK 状态码 |
| 密钥轮换 | ⚠️ 需协调 | 需通知电调开发者同步更新固件公钥，低频操作 |
| 远程管控 | ✅ 完整 | 续期/撤销/升级功能均支持远程下发 |

### 6.4 合规可行性

| 维度 | 评估 | 说明 |
|------|:----:|------|
| 密码学合规 | ✅ | P-256、AES-256、SHA-256 均为 NIST/FIPS 推荐算法 |
| 数据隐私 | ✅ | 仅传输设备 ID 和产品信息，无个人隐私数据 |
| 出口管制 | ✅ | AES-256 已从美国出口管制清单放宽（EAR §740.17）|

## 七、安全性分析（重点）

> 详细密码学论证请参阅配套文档 [安全性深度解析.md](安全性深度解析.md)

### 7.1 安全属性评级

| 安全属性 | 评级 | 关键技术 |
|---------|:----:|---------|
| Token 不可伪造 | 🔒🔒🔒 极强 | ECDSA P-256（128-bit 安全）|
| 传输机密性 | 🔒🔒🔒 极强 | AES-256-GCM（$2^{256}$ 密钥空间）|
| 前向保密（PFS） | 🔒🔒🔒 极强 | ECDH 临时密钥，历史报文永远无法解密 |
| 防重放攻击 | 🔒🔒🔒 极强 | 临时密钥对 + nonce + 时间戳三重绑定 |
| 防设备间 Token 复用 | 🔒🔒🔒 极强 | Token 内嵌 `esc_device_id`，强制核验 |
| 固件逆向防御 | 🔒🔒🔒 极强 | 电调仅有公钥，逆向无增益 |
| 传输完整性 | 🔒🔒🔒 极强 | GCM 认证标签（128-bit AEAD）|
| RK3506 物理防护 | 🔒🔒 强 | 取决于 TPM/OTP/OP-TEE 实施质量 |

### 7.2 防御纵深示意

```
攻击者需要同时突破所有层次才能伪造 Token：

  层次 0：获取 RK3506 签名私钥
              ↓ 失败（TPM/OTP/OP-TEE 保护）
  层次 1：破解 ECDH 临时密钥
              ↓ 失败（ECDLP 困难问题，$2^{128}$ 安全）
  层次 2：破解 AES-256-GCM 加密
              ↓ 失败（$2^{256}$ 密钥空间）
  层次 3：伪造 ECDSA-P256 签名
              ↓ 失败（等同于解决 ECDLP）
  层次 4：伪造目标设备 ID
              ↓ 失败（硬件 UID/OTP 不可伪造）

  任意单层失效，后续层次仍提供兜底保护。
```

### 7.3 与已知攻击的对应关系

| 已知攻击类型 | 本方案对应防御 | 是否有效 |
|------------|--------------|:-------:|
| 串口被动监听 | AES-256-GCM 全程加密 | ✅ |
| 中间人主动插入 | GCM 认证 + ECDSA 验签 | ✅ |
| Token 重放攻击 | 每次新 session_key + nonce | ✅ |
| Token 拷贝到其他设备 | device_id 绑定 + 校验 | ✅ |
| 暴力猜测 Token | 签名覆盖所有字段，无法通过 | ✅ |
| 序列号类软件破解 | 无序列号，纯密码学验证 | ✅ |
| 时钟回拨延长有效期 | RTC Tamper 检测 + issue_ts 辅助 | ⚠️ 中 |
| RK3506 物理提取私钥 | OP-TEE / TPM 保护 | ✅（依实施）|

### 7.4 架构选型对比：传统双向互认证 vs 当前单向设计

这是本方案最重要的架构决策之一，需要从商业化防破解的核心诉求出发理解其合理性。

#### 7.4.1 两种架构的对比示意

```
传统双向互认证（mTLS / GitHub SSH 模式）：

  RK3506（授权方）                         电调（被授权方）
  持有：RK_private_key                     持有：ESC_private_key  ← 关键区别
  持有：ESC_public_key（注册的）            持有：RK_public_key（内置的）
       │                                        │
       │── 挑战：random_challenge ─────────────►│
       │                                        │  用 ESC_private_key 签名
       │◄─ 响应：Sign(ESC_priv, challenge) ────│
       │                                        │
       │  验证：Verify(ESC_pub, response) ✓     │  "我确认你是合法电调"
       │  用 RK_private_key 签名 Token          │
       │── Token + Sign(RK_priv, token) ────────►│
       │                                        │  验证：Verify(RK_pub, token) ✓

当前 secret_com 单向设计：

  RK3506（授权方）                         电调（被授权方）
  持有：RK_private_key（唯一私钥）          持有：RK_public_key（验证公钥，非机密）
       │                                        │
       │── HELLO_REQ（临时 ECDH 公钥）─────────►│
       │◄─ HELLO_RSP（临时 ECDH 公钥）─────────│  ← 无签名，无身份证明
       │                                        │
       │  用 RK_private_key 签名 Token          │
       │── TOKEN_PUSH（AES-GCM 加密）──────────►│
       │                                        │  Verify(RK_pub, token) ✓
       │                                        │  核查 device_id 一致性
```

#### 7.4.2 核心差异一：ESC 固件被 dump 后的后果（最关键）

```
场景：攻击者购买一台电调，通过 JTAG/SWD 完整 dump 固件

  ┌──────────────────────────────────────────────────────────────────┐
  │  传统双向互认证：                                                 │
  │    dump 固件 → 获取 ESC_private_key                              │
  │    → 可编写程序冒充合法电调向任意 RK3506 完成身份验证             │
  │    → RK3506 以为在和真实电调通信，为其颁发 Token                  │
  │    → 攻击者持有合法 Token，可注入任意仿冒设备运行         ✗✗✗    │
  │                                                                  │
  │  当前 secret_com 单向设计：                                       │
  │    dump 固件 → 只获得 RK_public_key（公钥，本就非机密）           │
  │    → 公钥无法签名，无法伪造任何 Token                             │
  │    → dump 行为对攻击者完全零收益                          ✓✓✓    │
  └──────────────────────────────────────────────────────────────────┘

  结论：对于"防固件逆向仿制"这一商业核心诉求，
        电调不持有私钥在安全性上是正确且更优的选择。
```

#### 7.4.3 核心差异二：RK3506 能否验证对面是真实电调

```
传统双向互认证：
  RK3506 通过验证 ESC_private_key 的签名，可密码学证明
  "对面设备确实持有出厂烧录的私钥，是真实电调"            ✓

当前 secret_com（单向）：
  RK3506 无法密码学证明对面是否是真实电调
  攻击者可用自定义程序模拟 HELLO_RSP，假冒任意 device_id  ⚠️

  补偿机制：白名单 + Token 设备绑定
  ┌────────────────────────────────────────────────────────────┐
  │  攻击者假冒 device_id=0xABCD1234 发起握手                  │
  │       ↓                                                    │
  │  RK3506 查白名单：0xABCD1234 是否已注册？                  │
  │  ├── 不在 → 直接拒绝颁发（kAuthDenied），攻击终止  ✓      │
  │  └── 在   → 颁发绑定了 0xABCD1234 的 Token               │
  │             真实电调 0xABCD1234 照常工作，不受影响          │
  │             Token 颁发本身对授权体系无实质危害       ✓      │
  └────────────────────────────────────────────────────────────┘

  实际危害度评估：中等偏低
  攻击者即使冒充成功，也只是让 RK3506 为真实设备多颁发了一次 Token，
  无法将该 Token 用于其他设备（device_id 绑定），无法伪造新 Token。
```

#### 7.4.4 核心差异三：量产密钥管理复杂度

```
传统双向互认证的量产挑战：
  每台电调出厂必须：
  ① 为每台设备单独生成唯一密钥对（共 N 台 = N 次密钥生成）
  ② 私钥安全烧入每台电调受保护 Flash（OTP / TrustZone）
  ③ 对应公钥上传公司数据库，与设备序列号绑定
  ④ 公钥注册至 RK3506 白名单
       ↓
  生产节拍慢、SOP 复杂、一台失误则该台永久无法授权

当前 secret_com 的量产优势：
  全部电调出厂只需：
  ① 同一份固件，编译时已内置公钥（所有型号共享同一公钥常量）
  ② 生产线扫码录入设备 ID 至 RK3506 白名单
  ③ 无需为每台设备单独执行任何密码学操作
       ↓
  生产节拍快、SOP 简单、大规模量产友好
```

#### 7.4.5 核心差异四：私钥泄露的爆炸半径

| 泄露场景 | 传统双向互认证 | 当前单向设计 |
|---------|:------------:|:-----------:|
| 某台电调固件被 dump | 该台设备身份被冒充（每台独立密钥时）| ✅ 仅获得公钥，零影响 |
| 同型号共用 ESC 密钥被 dump | ❌ 同型号**全部**设备身份被冒充 | ✅ 仍零影响 |
| RK3506 私钥泄露 | 授权体系失效（与单向方案相同）| 授权体系失效 |
| 高价值私钥所在位置 | ❌ 在**易被攻击的电调**中 | ✅ 在**可保护的 RK3506** 中 |

**关键洞察：** 传统方案若为降低量产成本而采用"同型号共用 ESC 密钥"，其爆炸半径与本方案最差情况相同，却还多了"电调固件泄露即全线失效"的风险；若采用"每台独立密钥"，量产成本极高。本方案在两者之间找到了最优平衡点。

#### 7.4.6 商业防破解场景的威胁优先级分析

```
对商业化电调产品，威胁优先级从高到低：

  ① 【最高】攻击者 dump 电调固件、逆向、批量仿制
           当前方案：dump 只得公钥，无法仿制授权        ✓✓✓
           双向互认：dump 得到私钥，可无限克隆身份       ✗✗✗

  ② 【高】 攻击者截获串口通信、提取 Token 复用
           当前方案：AES-GCM 加密 + device_id 绑定      ✓✓✓
           双向互认：防御效果基本相同（均依赖加密）        ✓✓

  ③ 【中】 攻击者物理攻击 RK3506 提取私钥
           两种方案均依赖 TPM/OTP 物理保护，无本质差异

  ④ 【低】 攻击者伪造 Token 注入
           两种方案均依靠 ECDSA 签名，无本质差异         ✓✓✓

  最高优先级威胁（固件逆向仿制）：当前方案明显优于传统双向互认证
```

#### 7.4.7 综合对比矩阵

| 维度 | 传统双向互认证（mTLS 风格）| 当前 secret_com 方案 |
|------|:------------------------:|:-------------------:|
| ESC 固件被 dump 的后果 | ❌ 私钥泄露，可冒充身份 | ✅ 仅公钥，零收益 |
| RK3506 密码学验证 ESC 真实性 | ✅ 可证明 | ⚠️ 仅白名单 + device_id |
| 前向保密（PFS） | ❌ 通常无 | ✅ ECDH 临时密钥 |
| Token 不可伪造 | ✅ | ✅ |
| 量产密钥管理复杂度 | ❌ 每台独立密钥 | ✅ 全型号相同公钥 |
| 高价值私钥所在位置 | ❌ 在易攻击的电调端 | ✅ 集中在 RK3506 |
| 跨设备 Token 复用防御 | ✅ | ✅ device_id 绑定 |
| 固件逆向 → 授权克隆 | ❌ 可行 | ✅ 不可行 |
| 电调端实现复杂度 | ❌ 需签名能力 | ✅ 仅验签，约 200 行 |

#### 7.4.8 可选加强路径（不引入普通私钥）

若未来需要增强 RK3506 对电调真实性的验证，推荐方向**不是**给电调加普通私钥，而是：

```
方向 A：芯片 PUF（物理不可克隆函数）
  部分 MCU（如 STM32L5、NXP LPC55S）内置 PUF 模块
  PUF 根据芯片物理特征生成唯一密钥，无法从固件 dump 中提取
  RK3506 在握手时发起 PUF 挑战，电调用 PUF 派生密钥响应
  → 既保留"无普通私钥"优势，又增加了硬件级设备身份证明

方向 B：专用安全芯片（ATECC608B / SE050，成本约 ¥3-5/片）
  在电调 PCB 上增加一颗安全芯片
  私钥存于安全芯片内部，固件 dump 无法提取任何密钥材料
  安全芯片仅暴露签名接口，MCU 固件不直接接触私钥
  → 真正实现双向互认证，且从根本上解决私钥 dump 问题

方向 C：维持当前方案（当前阶段推荐）
  已满足商业防仿制的核心需求
  量产成本最低，实施最简单
  在 ESC 端无私钥的情况下已实现充分安全强度
```

**当前阶段建议：** 维持现有单向设计，在方向 B（安全芯片）成本可接受时作为未来升级选项纳入产品路线图。

## 八、风险与挑战

### 8.1 主要风险项

| 风险 | 严重程度 | 概率 | 缓解措施 |
|------|:--------:|:----:|---------|

| RK3506 被克隆（MITM）| 🟠 高 | 极低 | Token ECDSA 签名仍需私钥；克隆无法伪造 Token |
| mbedTLS 安全漏洞 | 🟡 中 | 低 | 定期更新依赖库；持续关注 CVE |

### 8.2 量产实施要点

```
生产线安全要点：

  1. 私钥生成环境
     □ 在隔离的内网机器上生成密钥对
     □ 生成完毕立即删除临时文件
     □ 私钥传输使用加密通道（不得通过 USB 明文拷贝）

  2. RK3506 初始化
     □ 私钥写入 OP-TEE 安全存储 或 TPM 2.0 密封存储

     □ 封闭调试接口（禁用 JTAG/SWD debug access）

  3. 电调固件
     □ 编译时将正确公钥嵌入 kAuthVerifyPublicKey 数组
     □ 通过 SWD 写入后锁定 Flash 读保护位（RDP Level 1/2）
     □ 禁用串口调试输出（防止泄露内部状态）
```

| 维度 | **本方案（RK3506+ECDSA）** | 纯软件序列号 | 硬件加密狗 | 在线授权验证 | 对称密钥方案 |

| 嵌入式适配 | ✅ | ✅ | ❌ | ❌ | ✅ |
| 远程吊销 | ✅ | ❌ | ❌ | ✅ | ❌（难）|
| 前向保密 | ✅ | ❌ | ❌ | ❌ | ❌ |

### 9.2 本方案独特优势

**相比纯软件序列号：**
序列号方案一旦被逆向分析，可以生成无限合法序列号；本方案无私钥则无法生成任何合法 Token，密码学保证不可绕过。

**相比硬件加密狗：**
加密狗需要额外物理接口（USB/串口），且成本高、体积大，不适合嵌入式 MCU 场景；本方案利用已有串口，无需额外接口。

**相比在线授权验证：**
在线验证要求设备联网，电调通常不联网（工业/无人机场景），且网络故障会导致设备无法工作；本方案完全离线可用。

**相比对称密钥方案：**
对称方案（如 AES 密钥烧入设备）只要一台设备被攻破就能克隆无限设备；本方案电调只有公钥，攻破任意数量电调对攻击者零收益。

### 9.3 与传统双向互认证（mTLS / SSH 风格）的专项对比

传统双向互认证是工业界成熟的设备身份验证模式（GitHub SSH、TLS 客户端证书均采用此模式），常被提出作为替代方案，以下是针对本产品场景的专项分析。

| 对比维度 | 传统双向互认证 | **本方案（单向私钥）** | 说明 |
|---------|:------------:|:--------------------:|------|
| ESC 固件 dump 后果 | ❌ 私钥泄露→可无限克隆 | ✅ 仅公钥→零收益 | 最核心优势，从架构上消除固件逆向威胁 |
| 量产密钥管理 | ❌ 每台独立密钥，SOP 复杂 | ✅ 全型号同一公钥 | 大幅降低量产成本和出错率 |
| 前向保密（PFS） | ❌ 通常不具备 | ✅ ECDH 临时密钥 | 历史报文被录制后也永远无法解密 |
| 高价值私钥位置 | ❌ 在易攻击的电调端 | ✅ 集中在 RK3506 | 私钥管理集中化，保护成本低 |
| 电调真实性密码学验证 | ✅ 可证明 | ⚠️ 白名单+device_id | 本方案的唯一劣势，可用 PUF/安全芯片补强 |
| 电调端实现复杂度 | ❌ 需签名能力 | ✅ 仅验签，~200行C | 对已量产固件改动极小 |
| 单台被破 → 全线失效 | ⚠️ 共用密钥时会 | ✅ 永远不会 | 电调侧无私钥，无爆炸半径 |

**结论：** 对于"商业电调防仿制克隆"这一核心商业诉求，传统双向互认证的主要劣势恰好集中在最高威胁场景（固件逆向）和量产成本上。本方案在这两点上明显占优，代价是 RK3506 无法密码学证明对面电调的真实性——而这一缺口在白名单机制下风险可控，并有 PUF/安全芯片的可选升级路径。

## 十、实施建议

### 10.1 分阶段实施路线

```
阶段一（1-2 个月）：核心功能验证
  □ 搭建 RK3506 开发环境，集成 secret_com 库
  □ 在开发板上完成 ECDH + AES-GCM + ECDSA 联调
  □ 选取 1-2 款在售电调型号接入 C API（150 行代码）
  □ 内部测试：覆盖正常授权、拒绝授权、Token 过期等场景
  □ 输出：功能验证报告

阶段二（1 个月）：安全加固与量产准备
  □ RK3506 私钥存储方案落地（OP-TEE 或 TPM 2.0）
  □ 生产线密钥写入流程规范化（SOP 文档）
  □ 电调 Flash 读保护位配置
  □ 远程管控接口（TLS 1.3 + HMAC 签名指令）联调
  □ 输出：量产工艺规范文档

阶段三（1 个月）：小批量试产与客户验证
  □ 选取 1-2 家合作客户进行封闭测试
  □ 收集反馈，完善异常处理和错误提示
  □ 性能基准测试（授权握手耗时、Flash 占用）
  □ 输出：测试报告 + 客户验收

阶段四（持续）：全面推广与运营
  □ 扩展至所有电调产品线
  □ 建立密钥轮换 SOP
  □ 建立授权审计日志系统
  □ 定期 mbedTLS 安全更新
```

### 10.2 关键决策点（需内部讨论确认）

| 决策项 | 选项 A | 选项 B | 建议 |
|--------|--------|--------|------|
| RK3506 私钥存储 | OP-TEE 安全世界 | Linux TPM 2.0 | OP-TEE（更强）|
| 密钥粒度 | 全产品线共用一对密钥 | 每产品型号独立密钥 | 建议按型号隔离 |
| 初始授权模式 | 仅永久买断 | 永久 + 订阅并行 | 先上永久，再扩展订阅 |
| HELLO 阶段加强认证 | 当前设计（无签名）| 增加双向认证 | 当前设计已够用，可后期加强 |
| 电调 RTC 要求 | 仅订阅模式需要 | 所有模式均配置 | 按需配置，减少改动 |

## 十一、结论与建议

### 总体评估

| 维度 | 评分 | 说明 |
|------|:----:|------|
| 技术成熟度 | ⭐⭐⭐⭐⭐ | 所有依赖均为成熟工业标准库 |
| 安全强度 | ⭐⭐⭐⭐⭐ | 128-bit 等效安全，多层纵深防御 |
| 实施可行性 | ⭐⭐⭐⭐ | 电调改动极小，主要工作在 RK3506 侧 |
| 运营便利性 | ⭐⭐⭐⭐ | 远程管控完善，但密钥轮换需协调 |
| 成本合理性 | ⭐⭐⭐⭐ | 硬件成本增加有限，长期商业价值显著 |
| **综合评分** | **⭐⭐⭐⭐⭐** | **建议采纳** |

### 建议采纳理由

1. **安全性行业领先**：算法选型（P-256 + AES-256-GCM + HKDF）均为当前密码学最佳实践，达到 128-bit 安全强度，前向保密设计优于大多数商业授权方案。

2. **对现有产品影响极小**：电调端仅需 150～200 行 C 代码，不改变现有功能，接入成本极低。

3. **授权策略灵活**：永久买断和订阅续费双模式，功能分级（kFeatureBasic/Advanced/Premium），满足不同商业场景。

4. **远程可控**：公司后台可随时更新白名单、续期、撤销，无需物理接触设备。

5. **技术债务低**：密码学后端（CryptoProvider）和传输层（Transport）均为可替换接口，未来可无缝切换 wolfSSL、HSM 或 BLE 传输，无需重写授权逻辑。
