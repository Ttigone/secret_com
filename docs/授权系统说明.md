# secret_com 授权系统说明

> 版本 2.0 · 2026 年 3 月
> 适用读者：电调设备开发者、RK3506 授权设备集成者、系统架构评审人员

---

## 一、产品定位与设计原则

本公司的主要产品是**电调设备（ESC）**。电调出售时，随附一个配套的 **RK3506 授权设备**，两者成对销售给客户。

**核心设计原则：电调少做事，RK3506 多做事。**

电调是已成熟开发的嵌入式产品，对其固件的改动应尽可能少，仅增加一个轻量级的凭证验证入口。所有授权管理逻辑、密码学运算、白名单管理、远程升级等功能，全部集中在 RK3506 上实现，便于后续迭代升级而不影响电调固件。

---

## 二、角色职责划分

### 电调（ESC）——轻量被动方

电调只需做以下三件事：

1. 在固件中**烧录一个验证公钥**（65 字节，只读，出厂写入一次，不属于机密信息）。
2. 上电后**等待 RK3506 通过串口推送授权 Token**。
3. 用内置公钥**验证 Token 的签名**，验签通过则正常工作，否则拒绝启动。

电调不持有私钥，不做授权决策，不管理白名单，不参与任何业务策略判断。

### RK3506（授权设备）——授权主控方

RK3506 负责所有授权相关的主动操作：

1. 持有**签名私钥**，存储在受保护存储区。
2. 管理**设备白名单**（哪些电调设备 ID 允许被授权，允许哪些功能）。
3. **生成并签名授权 Token**，将电调设备标识、授权参数打包并签名。
4. 通过串口将 Token **主动推送给电调**。
5. 支持**远程策略更新**：公司可通过网络向 RK3506 下发新的白名单、授权策略乃至新的密钥，无需触碰电调固件。

| 设备 | 持有密钥 | 主要操作 | 复杂度 |
|------|----------|----------|--------|
| 电调（ESC） | 验证公钥（65 字节，只读，非机密） | 验证 Token 签名、门控业务逻辑 | 极低，约 200 行 C 代码 |
| RK3506（授权设备） | 签名私钥（32 字节，受保护） | 生成 Token、管理白名单、远程升级 | 完整实现，集中维护 |

## 三、整体授权流程

### 3.1 概述

```
    公司后台                RK3506（授权设备）                  电调（ESC）
       |                          |                               |
[出厂阶段]                         |                               |
       |-- 私钥写入 RK3506 ------->|                               |
       |-- 公钥提供给电调开发者 ---------------------------------->|（烧录进固件）
       |                          |                               |
[运行阶段]                         |                               |
       |                          |<-- 上电，等待 Token 推送 ------|
       |                          |                               |
       |                 生成授权 Token                            |
       |                 绑定电调设备 ID + 功能 + 时间戳           |
       |                 用私钥 ECDSA 签名                         |
       |                          |                               |
       |                          |-- [串口] 发送 Token --------->|
       |                          |                               |
       |                          |        用内置公钥验证签名      |
       |                          |        检查设备 ID 一致性      |
       |                          |        验证通过 → 正常工作    |
       |                          |<-- ACK (OK/FAIL) --------------|
```

### 3.2 串口通信详细流程

RK3506 主动发起，电调被动接收。流程分两个阶段：

**阶段一：建立加密信道（防止 Token 在串口上被截获）**

```
RK3506（主动方）                                  电调（被动方）
      |                                                 |
      |-- [0x01] HELLO_REQ --------------------------->|
      |   rk_eph_pub   (65 字节, 临时 ECDH 公钥)       |
      |   rk_timestamp (8 字节, Unix 秒)                |
      |   rk_random    (16 字节, 随机数)                |
      |                                                 |
      |                               生成一次性临时密钥对
      |                               生成 esc_random
      |<-- [0x02] HELLO_RSP -----------------------------|
      |   esc_eph_pub  (65 字节)                        |
      |   esc_random   (16 字节)                        |
      |                                                 |
      | 双方独立计算 session_key                         |
      | ECDH(rk_eph_priv, esc_eph_pub)                  |
      | HKDF-SHA256(shared, rk_random||esc_random)      |
```

电调在 HELLO_RSP 中生成并提供一次性临时 ECDH 公钥，不需要使用设备静态私钥，不做任何签名，仅完成密钥协商即可。这是对电调密码学能力的唯一要求。

**阶段二：Token 下发与验证**

```
RK3506（主动方）                                  电调（被动方）
      |                                                 |
      |-- [0x03] TOKEN_PUSH (AES-256-GCM 加密) ------->|
      |   nonce     (12 字节)                           |
      |   密文载荷  (Token 明文加密)                    |
      |   GCM-tag   (16 字节)                           |
      |                                                 |
      |                               AES-GCM 解密      |
      |                               ECDSA 验签（公钥）|
      |                               校验设备 ID       |
      |                               校验有效期        |
      |<-- [0x04] TOKEN_ACK -----------------------------|
      |   status (1 字节)                               |
      |-- [0x06] DISCONNECT --------------------------->|
      |                                                 |
      |                               授权标志置位       |
      |                               电调开始工作       |
```

## 四、授权 Token 结构

### 4.1 Token 明文布局

Token 明文固定为 60 字节，后跟可变长度签名：

| 偏移 | 长度 | 字段 | 说明 |
|------|------|------|------|
| 0 | 4 | esc_device_id | 被授权电调的设备唯一 ID（uint32 小端） |
| 4 | 1 | feature_flags | 功能位掩码（见 4.3 节） |
| 5 | 1 | license_type | 授权类型（见 4.4 节） |
| 6 | 2 | padding | 补零对齐 |
| 8 | 8 | issue_timestamp | 颁发时间（Unix 秒，uint64 小端） |
| 16 | 8 | expiry_timestamp | 过期时间（0 表示永不过期） |
| 24 | 4 | remaining_uses | 剩余可用次数（次数限制类型有效，否则为 0） |
| 28 | 32 | product_name | 产品名称（ASCII，以 NULL 结尾） |
| 60 | 1 | sig_len | 签名字节数（不超过 72） |
| 61 | ≤72 | signature | ECDSA-SHA256 签名（DER 编码） |

### 4.2 签名计算

RK3506 在生成 Token 时，用自身持有的签名私钥计算：

```
signed_data = SHA256(
    "secret_com_token_v1\0"    (20 字节，域分隔符，防跨协议攻击)
    || esc_device_id            (4 字节)
    || feature_flags            (1 字节)
    || license_type             (1 字节)
    || padding                  (2 字节)
    || issue_timestamp          (8 字节)
    || expiry_timestamp         (8 字节)
    || remaining_uses           (4 字节)
    || product_name             (32 字节)
)                               // 共 80 字节输入 → SHA256 输出 32 字节

signature = ECDSA-P256-Sign(rk3506_private_key, signed_data)
```

电调收到 Token 后，用固件内置的**验证公钥**执行：

```
result = ECDSA-P256-Verify(esc_builtin_public_key, signed_data, signature)
```

验证通过，且 `esc_device_id` 与本机一致，且未过期，则 `g_esc_authorized = 1`，允许工作。

### 4.3 功能位掩码（feature_flags）

| 位值 | 名称 | 说明 |
|------|------|------|
| 0x01 | kFeatureBasic | 基础功能 |
| 0x02 | kFeatureAdvanced | 扩展算法/参数集 |
| 0x04 | kFeaturePremium | 完整高级功能解锁 |
| 0xFF | kFeatureAll | 全部功能 |

### 4.4 授权类型（license_type）

| 值 | 说明 |
|----|------|
| 0x00 | 永久授权，不过期 |
| 0x01 | 时间限制，expiry_timestamp 生效 |
| 0x02 | 次数限制，remaining_uses 生效 |
| 0x03 | 试用授权 |

## 五、帧格式规范

所有串口消息统一使用以下帧格式，所有多字节整数均为**小端序**。

```
偏移  长度  字段
 0     1   起始字节 0  固定值 0xAA
 1     1   起始字节 1  固定值 0x55
 2     1   MsgType     消息类型（见下表）
 3     2   SeqNum      序列号，uint16 小端，超过 0xFFFF 后回绕为 0
 5     2   PayloadLen  载荷字节数，uint16 小端，最大 512
 7     N   Payload     消息载荷
7+N    2   CRC16       CRC16-CCITT，覆盖字节 [0 .. 7+N-1]
```

帧总开销：9 字节（7 字节头 + 2 字节 CRC）。

CRC16 算法：多项式 0x1021，初始值 0xFFFF（CRC16-CCITT）。

### 消息类型定义

| 值 | 名称 | 方向 | 说明 |
|----|------|------|------|
| 0x01 | HELLO_REQ | RK3506 → 电调 | 握手请求，RK3506 临时公钥 + 随机数 |
| 0x02 | HELLO_RSP | 电调 → RK3506 | 握手响应，电调临时公钥 + 随机数 |
| 0x03 | TOKEN_PUSH | RK3506 → 电调 | AES-GCM 加密的授权 Token |
| 0x04 | TOKEN_ACK | 电调 → RK3506 | Token 验证结果（1 字节状态码） |
| 0x05 | HEARTBEAT | 双向 | 保活，载荷为空 |
| 0x06 | DISCONNECT | 双向 | 正常断开 |
| 0xFF | ERROR | 双向 | 协议错误，载荷为 1 字节错误码 |

### HELLO_REQ 载荷（89 字节）

| 偏移 | 长度 | 字段 |
|------|------|------|
| 0 | 65 | rk_eph_pub（临时 P-256 公钥，非压缩格式） |
| 65 | 8 | rk_timestamp（Unix 时间戳，小端 uint64） |
| 73 | 16 | rk_random（随机数） |

### HELLO_RSP 载荷（81 字节）

| 偏移 | 长度 | 字段 |
|------|------|------|
| 0 | 65 | esc_eph_pub（临时 P-256 公钥，一次性生成，不是设备静态密钥） |
| 65 | 16 | esc_random（随机数） |

### TOKEN_PUSH 载荷（加密封装）

| 偏移 | 长度 | 字段 |
|------|------|------|
| 0 | 12 | AES-GCM 随机 nonce |
| 12 | N | AES-256-GCM 密文（Token 明文加密） |
| 12+N | 16 | GCM 认证标签 |

### TOKEN_ACK 载荷（1 字节）

| 值 | 含义 |
|----|------|
| 0x00 | 验证通过，电调已授权 |
| 0x01 | ECDSA 签名验证失败 |
| 0x02 | Token 已过期 |
| 0x03 | 设备 ID 不匹配 |
| 0x04 | CRC 校验错误 |
| 0x05 | 帧格式错误 |

## 六、会话密钥派生

握手完成后，双方独立计算相同的会话密钥，用于加密 TOKEN_PUSH：

```
shared_secret = ECDH(本端临时私钥, 对端临时公钥)   // P-256 X 坐标，32 字节

session_key   = HKDF-SHA256(
                    IKM  = shared_secret,
                    salt = rk_random || esc_random,  // 32 字节
                    info = "secret_com_session_key_v1",
                    L    = 32                          // AES-256
                )
```

## 七、电调开发者集成指南

### 7.1 接入工作量说明

电调开发者只需完成以下工作，接入总代码量约 150～200 行 C 代码：

| 工作项 | 说明 | 难度 |
|--------|------|------|
| 烧录公钥常量 | 将公司提供的 65 字节公钥数组写入固件 | 极低 |
| 实现 2 个 UART 回调 | send / receive，对接已有 HAL 驱动 | 极低 |
| 调用 HandleOne | 在主循环或 RTOS 任务中调用一次即可 | 极低 |
| 实现业务门控 | 根据 `esc_is_authorized()` 决定是否启动电机 | 极低 |

### 7.3 设备 ID 的获取

电调需提供一个 32 位设备唯一 ID。RK3506 在生成 Token 时将此 ID 绑定其中，电调收到后校验一致性，防止 Token 被挪用到其他设备。

**优先方案：使用芯片硬件 UID**

| 芯片系列 | UID 寄存器地址 | 长度 |
|----------|----------------|------|
| STM32F4 | 0x1FFF7A10 | 96 位 |
| STM32F1 | 0x1FFFF7E8 | 96 位 |
| STM32G0/G4 | 0x1FFF7590 | 96 位 |
| STM32H7 | 0x1FF1E800 | 96 位 |
| GD32F4 | 0x1FFF7A10 | 96 位 |
| NXP LPC55Sxx | PUF/UUID 寄存器 | 128 位 |

读取并折叠为 32 位：

```c
static uint32_t esc_get_device_id(void)
{
    const uint32_t* uid = (const uint32_t*)0x1FFF7A10U; /* STM32F4 示例 */
    return uid[0] ^ uid[1] ^ uid[2];
}
```

**芯片无 UID 时的替代方案**

方案一（推荐）：工厂烧录唯一序列号

生产线上由上位机为每块电调分配唯一 32 位序列号，通过 SWD/JTAG 写入受保护 Flash 扇区：

```c
#define DEVICE_ID_FLASH_BASE  0x08070000U   /* 根据芯片型号调整 */
#define DEVICE_ID_MAGIC       0x44455649U   /* "DEVI" */

static uint32_t esc_get_device_id(void)
{
    const uint32_t* p = (const uint32_t*)DEVICE_ID_FLASH_BASE;
    if (p[0] == DEVICE_ID_MAGIC) return p[1];
    return 0xFFFFFFFFU; /* 未烧录时返回无效值，RK3506 拒绝生成 Token */
}
```

方案二：OTP 一次性可编程区域

部分芯片支持 OTP 区域工厂一次性写入并永久锁定：

```c
/* STM32F4 OTP 字节区域示例 */
static uint32_t esc_get_device_id(void)
{
    return *(const uint32_t*)0x1FFF7800U;
}
```

方案三：复用已有编号

若电调已有出厂编号（CAN 节点 ID、工厂条码序列号等），可直接使用，无需新增存储。

### 7.4 I/O 回调实现

```c
#include "secret_com/secret_com.h"

static int esc_uart_send(void* ctx, const uint8_t* data, size_t len)
{
    (void)ctx;
    /* 替换为实际的 HAL UART 发送函数 */
    HAL_StatusTypeDef r = HAL_UART_Transmit(&huart2,
                              (uint8_t*)data, (uint16_t)len, 50U);
    return (r == HAL_OK) ? (int)len : -1;
}

static int esc_uart_recv(void* ctx, uint8_t* buf, size_t max_len,
                          uint32_t timeout_ms)
{
    (void)ctx;
    HAL_StatusTypeDef r = HAL_UART_Receive(&huart2,
                              buf, (uint16_t)max_len, timeout_ms);
    if (r == HAL_OK)      return (int)max_len;
    if (r == HAL_TIMEOUT) return 0;
    return -1;
}
```

### 7.5 完整初始化代码

```c
#include "secret_com/secret_com.h"

static SecretComServer*  g_server     = NULL;
static volatile int      g_authorized = 0;

/* Token 验证通过回调 */
static void on_token_ok(const SecretComLicenseInfo* lic, void* ctx)
{
    (void)ctx;
    /* 校验 Token 中的设备 ID 与本机一致 */
    if (lic->device_id != esc_get_device_id()) return;
    g_authorized = 1;
}

void esc_auth_init(void)
{
    SecretComIoCbs io;
    memset(&io, 0, sizeof(io));
    io.send         = esc_uart_send;
    io.receive      = esc_uart_recv;
    io.connect      = esc_uart_connect;     /* 串口已常开时直接返回 1 */
    io.disconnect   = esc_uart_disconnect;
    io.is_connected = esc_uart_is_connected;

    g_server = SecretComServerCreate();

    /* 只传入公钥，私钥传 NULL，电调不持有私钥 */
    SecretComStatus s = SecretComServerInit(g_server, &io,
                                             kAuthVerifyPublicKey, NULL);
    if (s != kSecretComOk) { /* 初始化失败处理 */ return; }

    /* 注册 Token 验证成功回调 */
    SecretComServerSetTokenCallback(g_server, on_token_ok, NULL);
}

/* 裸机主循环轮询（每次主循环调用） */
void esc_auth_poll(void)
{
    if (!g_server) return;
    SecretComServerHandleOne(g_server, 100U); /* 100 ms 超时后返回 */
}

/* 业务门控接口 */
int esc_is_authorized(void) { return g_authorized; }
```

### 7.6 主循环集成

```c
int main(void)
{
    /* 系统初始化由 CubeMX/HAL 生成，此处省略 */

    esc_auth_init();

    while (1) {
        esc_auth_poll();       /* 等待 RK3506 推送 Token，非阻塞轮询 */

        if (esc_is_authorized()) {
            motor_foc_task();  /* 正常运行电机控制 */
        } else {
            motor_stop();      /* 禁止启动，等待授权 */
        }
    }
}
```

## 八、RK3506 授权设备集成指南

RK3506 是授权体系的核心，运行 Linux 系统，承担所有主动操作，使用 secret_com 库的 C++ API。

### 8.1 私钥管理

RK3506 持有的签名私钥是整个授权体系的信任根。

安全存储要求：
- 推荐存储于 Linux TPM、OP-TEE 安全存储或加密分区。
- 公司出厂前通过安全通道写入，不经过普通刷机流程。
- 私钥不对外暴露，不写入日志，不通过普通串口或网络接口传输。

私钥生成（由公司统一执行，每批次或每产品线生成一次）：

```bash
# 生成 P-256 密钥对
openssl ecparam -name prime256v1 -genkey -noout -out rk3506_signing.pem

# 导出 65 字节非压缩公钥（供电调固件烧录）
openssl ec -in rk3506_signing.pem -pubout -outform DER -out rk3506_pub.der
# rk3506_pub.der 偏移 [27..91] 为 65 字节公钥内容

# 或使用项目工具脚本生成 C 头文件
python3 tools/keygen.py --out rk3506_keys.h
```

### 8.2 白名单管理

白名单完全存储和管理在 RK3506 本地，电调无感知。白名单记录允许被授权的电调设备 ID 及对应的授权策略。

白名单条目结构：

```cpp
struct AllowEntry {
    uint32_t esc_device_id;    // 电调设备唯一 ID
    uint8_t  feature_flags;    // 允许的功能掩码
    uint8_t  license_type;     // 授权类型
    uint64_t expiry_timestamp; // 过期时间（0 = 永久）
    char     product_name[32]; // 产品名称
};
```

白名单来源：
- 出厂时根据订单自动生成并写入 RK3506 存储。
- 公司后台通过网络远程下发更新（见第九章）。

### 8.3 Token 生成与推送

```cpp
#include "secret_com/auth_client.h"
#include "secret_com/types.h"

void rk3506_authorize_esc(const char* uart_dev,
                           uint32_t    esc_device_id,
                           uint8_t     feature_flags)
{
    // 1. 创建串口 transport
    auto transport = std::make_unique<secret_com::SerialTransport>(
                         uart_dev, 115200);

    // 2. 配置密钥（RK3506 侧传入私钥）
    secret_com::CryptoConfig cfg;
    cfg.is_server = false;
    load_signing_private_key(cfg.device_private_key); // 从安全存储加载私钥

    // 3. 初始化推送端
    secret_com::AuthClient client;
    client.Initialize(std::move(transport), cfg);

    // 4. 构造授权请求
    secret_com::AuthRequest req;
    req.client_device_id = esc_device_id;
    req.feature_flags    = feature_flags;
    strncpy(req.product_name, "ESC-Pro-X1", sizeof(req.product_name));

    // 5. 推送 Token（异步，10 秒超时）
    client.RequestAuthorization(req,
        [](secret_com::Status s, const secret_com::LicenseInfo& lic) {
            if (s == secret_com::Status::kOk) {
                // 电调已成功验证 Token，记录授权日志
                log_auth_success(lic.device_id, lic.issue_timestamp);
            } else {
                // 授权失败，记录原因
                log_auth_failure(s);
            }
        },
        10000);
}
```

### 8.4 获取电调设备 ID

RK3506 在推送 Token 前，需要知道目标电调的设备 ID，有两种方式：

方式一：出厂配对时扫码录入，预存于白名单，无需运行时查询。

方式二：通过可选的轻量查询命令从电调读取：

```
RK3506 → 电调:  [0xAA][0x55][0x10][seq:2][0x00][0x00][CRC16]
电调 → RK3506:  [0xAA][0x55][0x11][seq:2][0x04][0x00][id:4B][CRC16]
```

## 九、远程升级与策略更新

RK3506 作为授权体系的主控方，所有后续升级均在 RK3506 上完成，**无需更新电调固件**。

### 9.1 可远程更新的内容

| 更新内容 | 说明 |
|----------|------|
| 授权白名单 | 新增、删除或修改允许配对的电调设备 ID 及功能策略 |
| Token 有效期策略 | 调整特定设备的授权时间或次数限制 |
| 授权软件升级 | 升级 RK3506 自身的 secret_com 库及业务逻辑 |
| 签名密钥轮换 | 在私钥周期性轮换时，更新 RK3506 存储的私钥（属于计划内操作，需同步向电调开发者分发新公钥） |

### 9.2 密钥轮换流程

签名密钥的轮换涉及电调端公钥的更新，属于低频计划内操作：

1. 公司生成新密钥对。
2. 提前通知电调开发者，将新公钥以固件更新包形式发布（电调仅需替换一个 65 字节常量数组）。
3. 电调开发者发布新固件并推送给客户。
4. 新固件部署完成后，公司通过网络将新私钥写入 RK3506。
5. 轮换完成，后续 Token 均使用新密钥签发。

## 十一、安全保障说明

### 11.1 私钥集中于 RK3506，电调无需保护密钥

签名私钥只存在于 RK3506。电调固件中只有公钥，公钥本身不属于机密信息。即使电调固件被完整读取，攻击者也无法伪造 Token，无法绕过授权。密钥管理、轮换、审计的全部复杂性集中在 RK3506 一处，对电调开发者透明。

### 11.2 Token 绑定设备 ID，防止跨设备复用

每个 Token 中包含目标电调的 `esc_device_id`，电调收到 Token 后校验此 ID 与本机一致。即使 Token 在传输过程中被截获，也无法用于其他电调设备。

### 11.3 会话加密保护传输内容

HELLO 握手阶段建立临时 ECDH 会话密钥，TOKEN_PUSH 使用 AES-256-GCM 加密，提供传输机密性和完整性保护，防止串口被被动窃听或主动注入。每次上电握手使用全新的临时密钥对和随机数，历史报文无法复用。

### 11.4 防 Token 伪造

Token 携带 ECDSA-SHA256 签名，签名覆盖所有字段，并含有域分隔符防止跨协议重用。没有 RK3506 私钥就无法生成合法 Token，即使攻击者知道完整协议格式也无法通过电调的验签。

### 11.5 防 Token 重放

Token 包含 `issue_timestamp` 和 `expiry_timestamp`。电调可拒绝时间戳过旧的 Token（如颁发时间超过阈值且未存储到本地）。对于需要断电重启仍有效的场景，使用 `kPermanent` 类型并由电调将 Token 存储于内部 Flash，下次上电直接从 Flash 加载验证，无需重新握手。

### 11.6 双重保护机制

TOKEN_PUSH 对传输层加密（AES-GCM），电调解密后再对 Token 内容进行签名验证（ECDSA），两层独立保护：即使会话密钥泄露，伪造的 Token 仍无法通过签名验证；即使 Token 被截获，加密也保护了其内容不被分析。

## 十二、编译与依赖

### 电调端编译要求

- 将 secret_com 库的 `.lib` 或 `.a` 加入工程
- 包含路径添加 `secret_com/include`
- mbedTLS 依赖模块：`mbedtls_ecdh`、`mbedtls_ecdsa`、`mbedtls_aes`、`mbedtls_gcm`（约 30 KB Flash，相比完整版减少约 40%，因无 EcdsaSign 操作）
- 目标芯片：Cortex-M4/M7 或同等级 MCU，推荐 UART 115200 bps
- 堆栈需求：约 1.5～2 KB

### RK3506 端编译要求

- 运行 Linux 系统，使用 C++17 标准编译
- 依赖 mbedTLS 2.16+ 或 3.x 完整版
- 依赖 pthread（内置后台线程）
- 构建命令：`cmake --build . --target secret_com`
